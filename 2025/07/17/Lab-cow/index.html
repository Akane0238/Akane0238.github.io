6.s081 Lab: Copy-on-Write Fork

发表于2025-07-17|更新于2025-07-17
|阅读量:

COW fork 任务背景

fork 出的子进程需要复制其父进程的整个地址空间，具体来说它要调用 uvmcopy() 复制整个页表，把原本的数据全部复制到新申请的物理页中。如果父进程占用的内存很大的话，那么复制就产生不小的开销了，一般情况下 fork 后会直接 exec 加载新的程序，这需要清理旧地址空间，刚复制的物理页和页表会被proc_freepagetable一一释放掉，之前的复制工作变成徒劳。

避免无效复制的方法就是延迟复制操作，等到真正需要的时候再从父进程中复制到进程空间里。Copy-on-Write fork 就是利用 page fault 实现了这种想法：在 fork 之初子进程将映射父进程的所有物理页（trapframe页是私有的），同时把 parent 和 child 的页表的页表项权限设置为只读 PTE_R。

当其中一个进程（子进程或父进程）要写入数据的时候，触发 page fault，内核 trap handler 为其分配物理页，并把原本的物理页数据复制一份到新的物理页中，重新执行前面造成非法访问的指令。在该进程的视角里这些内核的操作都是透明的（transparent）。

Implement copy-on-write fork

Your task is to implement copy-on-write fork in the xv6 kernel. You are done if your modified kernel executes both the cowtest and usertests -q programs successfully.

说实话，一开时觉得这个任务的需求很清晰明了，感觉实现起来应该是不难的，但是实际上手感觉难度高过前面做的 superpage ，有许多的细节需要注意。由于对这个想法比较感兴趣，我一开始并没有看实验书给的 hints，而是按照自己的想法去尝试实现的，所以碰到了不少问题。在下面的分析与编码中我第一个版本的实现会写出来分析，最终的实现思路也会写出来。

ver1. fork 复制页表

COW fork 实现的思路明确，因为我们可以从 fork 顺着 page fault 沿着抽象层次向下根据问题实现需求。

首先 fork 不能为子进程分配物理页，我们只需要把父进程的所有 PTEs 复制到子进程的页表中，也就是说把父进程所有的物理页映射到了子进程的地址空间内。这个过程中我们遇到第一个需求：对于原本可写（设置了PTE_W权限）的页，要在父进程和子进程的页表中设置为不可写。

那么问题来了，（Q1）当后面 page fault 发生时，怎么区分该虚拟地址是真的没有写权限还是 COW fork 需要复制物理页？

第一个解决方案是利用对物理页的引用计数 pprefcnt[i] （physical page reference count）进行判断。引用计数是 COW 的另一个需求，需要维护对一个物理页面所有进程的引用数，当 refcnt 为 0 时，我们才能将其释放到 freelist 。同时引用计数也带来了一个优化，在后面实现中会提到（xv6 book）。

由于 uvmcopy 具有通用性，可能存在复制页表也须复制物理页的需求，所以我一开始实现了一个类似的 cowcopy ， fork 用这个函数而非通用的 uvmcopy 。使用 mappages 构建映射后我们需要递增父进程物理页的引用数。

ver1. usertrap 处理页错误

对于 COW 复制的页，由于没有写权限，子进程（或父进程）尝试写入该虚拟地址时会发生 page fault，RISC-V 规定这种非法写入的 scause 为 0xf（15）。

因此接下来我们就要修改 usertrap() 对于 page fault 的处理逻辑，当写入 COW 页时需要分配实际的物理页并复制原本物理页的数据，再重新执行出错的写入指令（地址保存在了 trapframe->epc）。

对于非法写入的处理我们需要区分是否是 COW 页（Q1），在 pagefault_store() 的尝试里我通过物理页的引用计数 refcnt 是否大于 1 来确定该物理页是由多进程共享的，而我们的实现里只有 fork 会共享物理页，所以该页是 COW 页，需要我们执行复制逻辑。

基于引用计数进行 COW 页判断的 pagefault_store 实现存在很多细节的问题。当一个 COW 页的引用 refcnt 为 1 时，对应的进程如果发生写入的 page fault，我们无需重新根据这个物理页分配一个新的物理页了，直接修改 PTE 权限就可以（这就是前面提到的优化）。但是这个优化在上述的逻辑里实现不了：当一个进程首先触发 page fault，该物理页的 refcnt 变为了 1 ，此时另一个进程也尝试写入，但是逻辑判断该页不是 COW 页，而是一个私有的页，它本来就没有写入权限，这样的结果显然是错误的。要想修正我们需要添加额外的逻辑，同时还要维护好物理页的引用计数，这使得情况变得很复杂。

ver1. kalloc/kfree 底层物理页管理

关于 COW 页释放，我们需要一套 “bookkeeping” 策略来维护对物理页引用的计数：

对于每一个物理页（地址0x80000000~0x88000000），我们以 (pa-KERNBASE)/PGSIZE 作为引用计数数组的 index，同时为了保护 pprefcnt 这个全局对象的安全读写，需要一个自旋锁来实现互斥。

现在每个物理页相当于多了一个字段 pprefcnt 用来指示被页表的映射数。而接下来对于物理页的管理涉及到了一个设计上的问题（Q2）：在 kfree 中判断是否真的需要释放这个物理页（给上层抽象，无需改动其余涉及 kfree 的代码）or 实现 kfree 为真的释放这个物理页，那么就需要由上层来维护 refcnt，当引用为 0 时才实际调用？

  1. kfree 不再是“无条件释放”，而是“检查并释放”。上层代码（如 uvmunmap）一样在解除一个 PTE 映射后，无脑地调用 kfree(pa)。真正的释放决策由 kfree 内部做出。
  2. kfree 保持其原始语义：“立即、无条件地将一个页面放回空闲链表”。引用计数的检查和维护的责任完全交给调用者。

对此，Gemini 的工程实践推荐为修改 kfree 在内部检查 refcnt。引用计数是物理内存分配器 (kalloc.c) 的内部实现细节，上层模块不应该、也不需要关心它，这体现了封装和信息隐藏的设计原则。其次这么做能极大地减少了代码的修改量，否则我需要修改上层一系列调用这个接口的函数。

ver1. 中道崩殂


scause=1 (Instruction access fault) 发生在 init 进程中，可以确定是 init 进程的代码页 (.text 段) 的页表项 (PTE) 出了问题，导致它变得不可执行 (PTE_X 位被清除)。后面审查了代码发现是位运算写错了，导致 PTE 高地址的 PPN 被擦除。

修复了这个问题后我又遇到了很多 bug，有些是像上面的编码问题，但是更多的问题指向的是引用计数设计的不良。我一直在专注于修补 bug，奇怪的 panic 和调试的复杂让我最后放弃了第一次实现，决定重构代码。

ver2. PTE 引入 COW 扩展位

回到 usertrap() 处理 page fault 的问题（Q1）：fork 会把原本无论是否具有写权限的PTE设置为不可写，为了恢复写权限，我们要怎么“保存”原先的 PTE_W 权限呢？

解决方法二：hint 告诉我么可以利用 RISC-V PTE 的保留位（RSW），设置 COW page 位用来表明是否需要 copy：

  • 对于可写的页面，我们在清除 PTE_W 的同时，设置 PTE_C 位表示它原本是可写的。
  • 对于只读的页面，我们什么都不做，直接共享，它的 PTE 中没有 PTE_C 位。

引入了 COW 位后对于父进程和子进程的页表操作又添加了许多细节，需要我们仔细审计。

修改 usertrap 页错误逻辑处理，这里我们只需要处理写入 COW 页这一个 exception，对于其他页错误和异常，按照原本的逻辑执行就可以，所以要在 usertrap 添加判断逻辑。

helper function cowfault() 实际处理 COW 页错误，在这个设计下我们可以正确地实现优化。注意最后我调用了 kfree 递减原本物理页的引用计数，具体是否真的释放则由 kfree 自行判断。

实现过程中依旧遇到了很多细节问题，但是没有一一记录下来。最后的实现通过了 cowtest ，但是没有通过 usertests 中的 MAXVAplus 测试。

usertrap 处理非法地址访问（这也是 exception ）时没有判断是否超出 MAXVA ，导致后面的 walk() 在内核中又访问了非法的地址，导致 panic。在 usertrap 中增加边界条件的过滤后通过测试🎉

最后总结一下这个实验的过程。耗时大概10h左右，目前为止个人感觉这个实验是最难的，superpage 其次。COW fork 的想法很有意思，通过 page fault 来实现 “lazy allocation” ，本身实现是不难的，难在任务要求实现引用计数的优化，对计数不当的增减会出现很多意想不到的问题，只能慢慢 debug 找到根源（第一次尝试我都找不到为什么）。

想一想实际上引用计数也是必要的，不然无法判断一个物理页被 fork 出的很多进程共享后，什么时候需要释放这个物理页。对于每个进程来说这个物理页都不可写，每次 page fault 都会复制物理页并 unmap 映射，那么当最后一个进程解除映射后，这个物理页就无法追踪了，造成内存泄露。想要确定谁是“最后一个”，简单的解决方案就是实现对物理页的引用计数。
